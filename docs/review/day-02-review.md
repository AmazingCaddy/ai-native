# Day 2：Prompt、Context 与结构化输出复习材料

## 结构正确与内容正确

结构化行动项提取至少需要分别验证两个维度：

| 维度 | 典型失败 | 主要控制手段 | 指标示例 |
| --- | --- | --- | --- |
| 输出结构 | JSON 外出现代码围栏、字段缺失、类型错误 | API 结构化输出、JSON Schema、运行时校验、稳定错误 | JSON 可解析率、Schema 通过率 |
| 业务内容 | 把既有目标当作新任务、遗漏行动项、捏造负责人 | 明确任务边界、正反例、原文证据、固定评测集 | 误提取、遗漏、字段正确性、证据一致性 |

Schema 能约束字段、类型、枚举等输出形状，但不能证明字段值符合原文事实或业务规则。即使 Schema 通过率为 100%，应用仍需进行业务规则校验和内容评测。

内容正确性还需要两类机制：

- 在线校验：对每次结果检查原文证据、缺失字段、人员集合、日期解析、冲突和确认要求。没有负责人或日期证据时返回 `null`，高风险结果不得自动执行。
- 离线 Eval：使用人工标注的固定样例集统计误提取、遗漏、字段正确率和证据一致性，在 Prompt、模型或 Schema 变更后执行回归。

Eval 衡量系统在一组样例上的表现，不能替代当前请求的在线校验。

一句话记忆：Schema 验形状，在线校验验当前结果，Eval 验系统整体表现。

### 可审计证据设计

证据应是输入原文的逐字引文，而不是模型生成的理由或摘要。对行动项、负责人和截止日期分别设置证据字段；某字段非 `null` 时，其证据必须存在并能在原文中定位，否则拒绝该字段、设为 `null` 或标记 `needs_confirmation`。

日期等规范化值和原始证据应同时保留。例如将“下周五前”转换为 ISO 日期时，`dueDateEvidence` 仍保存原文。应用至少验证引文是否为原文子串；更完整的实现还会保存字符位置、说话人和时间戳，并检查证据与具体字段之间的关系。

证据存在不等于结论正确，高风险操作仍需业务校验或人工确认。

### 输出粒度与可验证性

更细粒度的模型输出通常能提供更多独立检查点：字段级证据可检查原文存在性，来源片段和说话人可检查出处，规范化值与原始证据并存可审计转换过程。

但字段并非越多越好。只增加有下游用途、能独立验证或失败后有明确处理方式的字段；否则会增加 Schema 复杂度、Token、延迟和故障面。能够由代码确定性计算的信息，例如逐字引文的字符位置，应由应用计算而不是要求模型猜测。模型自报的置信度也不能直接作为真实性证明。

原则：模型返回可校验的最小充分声明，应用使用原文、代码和权威数据完成验证。

推荐先确保输出可被程序可靠解析和校验，再改善业务准确性；两类指标始终分开报告。

## 如何验证一次修复

使用固定评测集做前后对照：先保存旧版本基线，然后每次只改变一个主要因素，在相同模型、输入、参数和评分方法下重跑。除了已知失败样例，还要使用未参与调试的新样例检查过拟合。

一次结构修复可以要求 JSON 可解析率和 Schema 通过率达到 100%。一次内容修复不能只看目标错误消失，还要同时确认误提取下降、遗漏没有恶化，并且正常、无任务、缺字段和信息冲突等回归样例仍通过。

不能把局部指标改善等同于整体有效：结构通过率提升只证明格式改善；误提取下降但遗漏上升，也不代表内容质量整体提高。

## 提高 Schema 通过率

1. 优先使用模型 API 提供的受约束结构化输出，并在请求参数中提交 Schema，而不只是用 Prompt 描述格式。
2. 设计简单、明确的 Schema：明确必填字段、`null`、枚举、数组元素类型，并拒绝未声明字段；减少不必要的嵌套和复杂联合类型。
3. 统一缺失值表示，避免省略字段、空字符串与 `null` 混用。
4. 在应用边界再次执行运行时 Schema 校验，禁止未校验结果进入下游。
5. 区分解析失败、截断、拒答和 Schema 校验失败；只进行次数有限的重试，失败后返回稳定错误。
6. 用评测集持续统计 JSON 可解析率、Schema 通过率和失败类型。

这些措施提高的是结构可靠性，不能保证字段值符合原文事实或业务规则。

## 主流模型 API 的支持现状

截至 2026-07-19，主流第一方闭源模型 API 多数已经提供某种 Schema 约束型结构化输出。已核验的 OpenAI、Anthropic、Google Gemini 和 Mistral 均支持；但这不是所有模型、版本、端点和第三方兼容服务的共同保证。

特别注意：JSON Mode 通常只保证生成有效 JSON，原生 Structured Outputs 才进一步约束结果符合所提供的 Schema。即使厂商支持 Structured Outputs，也通常只支持 JSON Schema 的一个子集，而且拒答、截断、流式输出、工具组合与托管平台可能有单独限制。

选型时应核查当前模型 ID、API 端点、支持的 Schema 关键字、与流式输出及工具调用的组合限制，并始终保留应用侧运行时校验。

来源：[OpenAI](https://developers.openai.com/api/docs/guides/structured-outputs)、[Anthropic](https://docs.anthropic.com/en/docs/build-with-claude/structured-outputs)、[Google Gemini](https://ai.google.dev/gemini-api/docs/structured-output)、[Mistral](https://docs.mistral.ai/studio-api/conversations/structured-output)。核验日期：2026-07-19。

## Tool Calling 与 Strict Tool Use

Tool Calling 是较早出现的基础能力：模型选择工具并生成参数。Strict Tool Use 是后来增加的结构可靠性能力：通过受约束生成等机制，让工具名称和参数符合支持的 Schema 子集。支持 Tool Calling 不等于支持 `strict`。

以 OpenAI 为例，Function Calling 于 2023-06-13 发布，Structured Outputs 的 `strict: true` 于 2024-08-06 发布。选型时应分别记录 `supports_tools` 和 `supports_strict_tool_schema`。

来源：[OpenAI Function Calling](https://openai.com/index/function-calling-and-other-api-updates/)、[OpenAI Structured Outputs](https://openai.com/index/introducing-structured-outputs-in-the-api/)、[Anthropic Structured Outputs](https://docs.anthropic.com/en/docs/build-with-claude/structured-outputs)。核验日期：2026-07-19。

## 不支持 Structured Outputs 时的工程兜底

应用应建立 `生成 → JSON 解析 → Schema 校验 → 有限修复重试 → 稳定失败或降级` 的接口边界。

- 使用 API 可提供的最强约束，如 JSON Mode 或普通 Tool Calling，但保留应用侧 Schema 校验。
- 简化 Schema，并在 Prompt 中明确单一 JSON 输出、缺失值规则和边界示例。
- 保存原始输出，只做确定性的格式清理，不静默补造业务字段。
- 将具体校验错误反馈给独立修复请求，限制重试次数；仍失败则返回稳定错误、转人工或使用经过评测的后备模型。
- 自托管模型可以在推理服务层使用 grammar/guided decoding 增加生成约束。
- 分别统计首次通过率、修复后通过率、重试次数、最终失败率、延迟和 Token 成本。

未经校验的模型输出不得进入数据库或工具调用；Schema 通过后仍需业务规则和证据校验。

## 稳定指令与动态 Context

稳定指令描述跨一批请求保持一致的任务定义、接口契约和安全边界，例如行动项定义、输出字段、缺失值规则和禁止猜测。它们应当版本化，修改后执行回归评测。

动态 Context 描述当前请求的事实和偏好，例如会议正文、当前日期、有效时区和本次输出语言。它们应按请求注入，并与稳定指令明确分隔。

判断位置不能只看它是“定义”还是“值”，还要看变化频率、控制者和信任边界。输出语言若允许逐次选择就是动态偏好，若产品固定为中文则可以成为稳定配置；用户时区可以长期保存，但调用模型时仍应注入本次请求的有效值。

### 一句话记忆

稳定规则做版本，动态事实按请求注入；位置由变化频率、控制者和信任边界共同决定。

### 工程落地

- 稳定指令存放在源码或受控配置中，使用开发者/系统消息层级，记录 Prompt 版本，并通过固定评测集回归。
- 稳定 Schema 作为代码对象提交给模型 API，并用于应用侧运行时校验；Schema 与 Prompt 的兼容变更一起发布。
- 会议正文、当前日期、有效时区和输出语言按请求注入，放入明确的数据结构或分隔区域。
- 动态输入视为待处理数据，不能覆盖稳定指令；日志应根据敏感性执行脱敏、截断或摘要。
- 稳定前缀可以缓存和复用，动态内容通常只能用于当前请求。

不要只把两段文字拼接在一起；消息优先级和数据边界也是稳定指令与动态 Context 的实际区别。

### 不同 Provider 的字段映射

“稳定指令 + 动态 Context”是通用设计概念，不是所有 LLM API 共享的线级消息格式。

- OpenAI Responses API：使用顶层 `instructions` 与 `input`，或在 `input` 数组中使用 `developer` 和 `user` 消息。
- Anthropic Messages API：稳定指令通常使用顶层 `system`，消息主要使用 `user` 和 `assistant` 角色。
- Gemini API：通常映射到 `systemInstruction` 和 `contents`。

跨厂商应用宜先定义统一内部结构，再由 Provider Adapter 映射到各厂商 API，避免业务代码直接绑定某一家的消息字段。

OpenAI 来源：[Message roles and instruction following](https://developers.openai.com/api/docs/guides/text#message-roles-and-instruction-following)。核验日期：2026-07-19。

## Prompt 很长但效果不好时

不要立即追加规则。先用固定评测集复现失败并保存完整输入、原始输出、Prompt 与模型版本；核对期望标签，再依次检查 Context 的完整性和冲突、输出截断、Schema 与应用后处理。确认问题确实位于 Prompt 后，再删除重复或冲突规则、明确优先级，并用代表性边界示例替代零散补丁。

每次只改变一个主要因素，同时观察误提取和遗漏，避免修好单个样例却让整体退化。较长 Prompt 不一定错误；必要规则可以保留，但必须无冲突、有优先级且经过评测。

### 一句话记忆

先用失败样例定位层次，再精简或修改 Prompt；不要用新规则掩盖未知根因。

## 完整 Prompt 的六个部分

1. 目标：定义任务及业务边界，例如什么才算行动项。
2. Context：提供本次请求的会议正文、当前日期、时区和语言。
3. 约束：规定不得猜测、缺失值处理、证据要求和排除项。
4. 输出：声明业务字段；具体类型和必填规则由 Zod/JSON Schema 约束。
5. 示例：优先使用正例和容易误判的边界反例。
6. 完成标准：输出前检查证据、缺失信息、遗漏、重复和排除项。

这些部分不必机械重复：目标说明做什么，约束说明不能怎么做，完成标准负责最后收口。

一句话记忆：目标定边界，Context 给事实，约束防越界，Schema 定形状，示例讲难点，完成标准做收口。

## 缺失与冲突的表达

- 缺失：字段使用 `null`，证据为 `null`，必要时通过 `confirmation_reason` 标记缺少负责人或日期。
- 冲突：最终字段仍使用 `null`，同时保留候选值及各自证据，并将 `needs_confirmation` 设为 `true`。

是否因缺失字段阻止执行由业务规则决定；Schema 负责明确表达状态。只有当文本或业务规则能够可靠确定最终修正时，才可以从冲突候选中选择一个值。

不要混用空字符串、`"未知"`、省略字段和 `null` 表达缺失，也不要在冲突时只保留 `null` 而丢失候选证据。

一句话记忆：缺失用 null，冲突用候选加证据；不确定性要显式保留，不能猜成事实。

### 冲突后的确认工作流

冲突对象应进入 `pending_confirmation`，保存所有候选值及证据，并暂停数据库写入、任务创建等副作用。系统向有权限的用户展示候选和证据，允许选择候选、填写其他值、拒绝或稍后处理。

解决后记录 `resolved_by`、`resolved_at`、原始候选、最终选择和备注；原始冲突作为审计历史保留。无人确认时保持待确认或过期，不能超时后自动猜测。

字段冲突解决不等于授权执行外部写操作。创建真实任务前仍应展示最终参数预览并获得执行确认。

## 行动项提取的初始评测集

第一版至少覆盖：单个完整行动项、多个交错行动项、空文本、无行动项、缺负责人和日期、相对日期、字段冲突、明确修正或取消、多人同名或指代不清、长文本噪声。

指标应拆分为：JSON 可解析率、Schema 通过率、行动项 Precision 和 Recall、负责人正确率、日期正确率、字段捏造次数、证据有效率、确认状态正确率、最终失败率、延迟、Token 和人工修改时间。

Precision 发现多提，Recall 发现漏提，两者不能互相替代。先由人确定期望结果，再运行模型；格式和字段尽量确定性评分，自然语言语义评分规则必须固定。

一句话记忆：用例按失败类型覆盖，指标按错误维度拆分；Precision 防多提，Recall 防漏提。

## Prompt、Context 与模型能力归因

- Prompt 问题：任务边界、优先级或输出要求不清。固定其他条件，只修改 Prompt 后重跑评测。
- Context 问题：必要事实未提供、被截断、含噪声或互相冲突。固定其他条件，只恢复或整理 Context 后重跑。
- 模型能力或适配问题：在相同 Prompt、完整 Context、Schema 和参数下，某模型在一组同类型样例上持续落后于对照模型。

归因实验每次只改变一个主要因素，并使用固定评测集。单条失败不足以证明模型能力问题，真实故障也可能跨越多个层次。

一句话记忆：Prompt 决定怎么做，Context 决定看见什么，模型决定能力上限；固定其余变量再定位。

### Prompt 与 Context 的术语边界

广义 Context 是模型一次推理可见的全部 Token，包括系统/开发者指令、用户消息、会话历史、检索资料和工具结果；因此 Prompt 是 Context 的一部分，也会占用 Context Window。

工程排障时常把 Prompt/Instructions 狭义地指任务规则，把 Context/Data 狭义地指本次事实材料。这是便于定位问题的概念划分，不代表二者在物理输入上分离。不同团队也可能把全部模型输入统称为 Prompt，讨论前应明确术语。

## 规则、Few-shot 与严格 Schema

- 明确规则定义目标、业务边界和优先级；规则过多可能冲突、变长并难以维护。
- Few-shot 是在 Prompt 中提供少量输入—输出示范，适合用正例与边界反例展示难以仅靠抽象规则说明的语义区别；它占用 Token，并可能对示例选择敏感或过拟合。
- 严格 Schema 约束字段、类型、枚举和缺失值表达，但不保证字段值真实；它受 Provider 支持的 Schema 子集限制。

三者通常组合使用：规则说明“缺失负责人不得猜测”，Schema 规定 `owner` 必须存在且类型为 `string | null`；Few-shot 则展示“保持既有目标”与“新增行动项”的边界。

一句话记忆：规则讲原则，Few-shot 讲边界，Schema 守结构；三者都要接受 Eval 检验。

## `/actions/extract` 最小接口合同

- 请求：会议正文、当前日期、时区和可选输出语言；先执行请求 Schema、空白、长度、日期与时区校验。
- 响应：行动标题、可空负责人和日期、字段级证据、确认状态、确认原因与冲突候选；缺失值统一使用 `null`。
- 处理链路：输入校验 → 组装版本化 Prompt → 带超时调用模型 → 解析 → Zod 校验 → 证据与业务校验 → 推导确认状态 → 保存 Trace → 返回。
- Trace：记录请求 ID、Provider、模型 ID、Prompt/Schema 版本、延迟、Token、重试、停止原因和错误码；敏感原文按策略脱敏或不落盘。
- 稳定错误：区分 `INVALID_REQUEST`、`MODEL_TIMEOUT`、`MODEL_OUTPUT_PARSE_FAILED`、`MODEL_OUTPUT_SCHEMA_FAILED` 和 `OUTPUT_GROUNDING_FAILED`，不对外泄露原始异常。
- 测试：覆盖 Route 输入、Service 业务校验、Model Adapter 失败分支、10 类评测回归和未确认写操作保护。

缺失或冲突通常是成功响应中的业务状态，而不是 HTTP 错误。只有系统无法提供符合合同的安全结果时才返回接口错误。

一句话记忆：先验输入，后调模型；先验结构，再验事实；全程留 Trace，错误不泄漏。

### 调用元数据

调用元数据描述一次模型调用的身份、配置、用量和状态，而不是业务正文或模型答案本身。最小字段包括请求/Trace ID、Provider 请求 ID、精确模型 ID、Prompt 与 Schema 版本、关键参数、开始时间、延迟、Token、重试次数、停止原因、状态和稳定错误码。

它用于故障复现、版本比较、成本与延迟分析、告警和审计。API Key、凭据和不必要的完整会议正文不得进入普通元数据日志；确需保存的原始输入输出应放入受控、脱敏且有保留期限的 Trace 内容区。

## 评测基础设施

行动项评测数据应包含稳定 Case ID、类别、输入、人工期望、标签和版本。评分器分别计算 Schema 与证据通过率、Precision、Recall、字段正确率、确认状态正确率和字段捏造次数，避免单一总分掩盖具体退化。

评测命令默认只做数据与评分规则 dry run；真实模型调用必须显式执行，并保存模型、Prompt/Schema 版本、参数、原始输出、延迟和 Token。没有真实调用时只能报告基础设施验证结果，不能声称模型达到质量阈值。

2026-07-19，最终版 `actions-v1` 通过 Floway 使用 `gpt-5.6-terra` 完成 10 条合成 Case。原始响应的 Schema 与证据通过率均为 100%，无负责人或日期捏造。修正一个标题语义等价导致的评分器假阴性并保留原始报告后，独立重评分为 10/10，Precision、Recall、字段与确认状态均为 100%。

该结论只适用于当前小型合成集。评分规则曾在观察输出后修订，后续必须增加 held-out Case 并重复运行；四个 Prompt 版本对照尚未完成。

## Zod 字段不变量

`owner`、`due_date` 及其证据字段应始终存在，缺失统一使用 `null`。当负责人或日期非 `null` 时，对应证据必须为非空字符串；值为 `null` 时，直接证据也为 `null`，冲突候选及证据放入候选数组。

Zod 负责字段、类型、非空和跨字段配对规则；证据是否确实来自会议原文，需要 Service 层结合原始输入校验。

一句话记忆：字段固定、缺失用 null；Schema 检查配对，Service 检查证据来源。

### 边界与常见误区

- 只在 Prompt 中要求输出 JSON，不等于 API 对输出执行了 Schema 约束。
- 运行时 Schema 校验是必要的应用边界，不能因为模型支持结构化输出就省略。
- 单次误提取不应直接归因于模型能力；先检查任务定义、Context、边界示例和评测标签。
- 结构正确不等于事实正确，也不等于业务可直接执行。

### 一句话记忆

Schema 管形状，评测管内容；先保证机器能读，再验证业务上读得对。
