# AI 技术判断矩阵

最后更新：2026-07-18

> 本矩阵记录主线项目的实际技术取舍。模板中的通用建议不能替代这里的项目证据。

| 技术/模式 | 解决的问题 | 使用条件 | 不使用条件 | 主要失败模式 | 质量/延迟/成本 | 替代方案 | 项目证据 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| TypeScript + Node.js 24 | 建立可测试的应用和实验工具 | 团队熟悉 TypeScript，需要快速迭代 API 与脚本 | 运行环境不支持 Node 24，或现有项目采用其他技术栈 | 类型与运行时不一致、依赖升级不兼容 | 高/低/中 | Python、Go | Day 1 类型检查、测试和构建通过 |
| Fastify | 提供轻量 API 骨架和请求注入测试 | 需要小型 Node API、快速健康检查和自动测试 | 只需要一次性 CLI，或必须复用既有 Web 框架 | 插件和生命周期理解错误 | 高/低/低 | Node HTTP、Express | `GET /health` 和自动测试通过 |
| 合成评测数据 | 在没有真实授权数据时验证工程链路 | 数据安全优先，真实资料尚未授权或脱敏 | 需要形成真实业务效果结论 | 场景过于简单、产生天花板效应 | 中/低/低 | 授权脱敏数据、公开基准 | 5 份文档、3 份会议记录、10 个问题；模型实验暴露区分度不足 |
| Codex CLI Adapter（历史，已移除） | 曾用于复用 Floway/Copilot 认证运行实验 | 仅用于解释 Day 1 历史结果 | 当前模型对比不再使用 | 用户级配置漂移、Agent 系统上下文影响结果 | 中/中/中 | Floway Responses API | 16/16 次历史调用成功；执行代码已于 2026-07-19 移除 |
| Floway Responses Adapter | 通过统一代理直接比较多个模型 API | Floway 提供 OpenAI-compatible endpoint、Key 和模型 ID | 代理不支持目标模型能力 | 不同模型兼容层差异、API 用量、把代理行为误判为模型能力 | 高/中/中 | 厂商原生 SDK、手工对比 | 三模型 24/24 调用成功；Terra 平均延迟和输出 Token 最低，完整边界见直连对比报告 |
| 统一条件模型对比 | 比较模型在相同任务与约束下的表现 | 任务、Prompt、数据、次数和评分标准可固定 | 样本过少却要做一般能力排名 | 天花板效应、把格式偏好误判为通用能力 | 中/中/中 | 各模型最优配置实验、公开基准 | GPT 8/8 满足标准；Claude 4/8，差异集中于格式与长度约束 |
| RAG | 在推理时补充频繁变化的外部知识 | 文档持续更新、答案必须有来源、知识可检索 | 问题来自行为、格式或基础能力不足 | 切块不当、漏召回、检索内容不支持答案 | 中高/中/中 | 长上下文、Tool Calling、微调 | Day 1 完成流程设计；计划 Day 6 用真实评测验证 |
| Reranker | 对初次召回的候选块精排 | Top-N 候选中包含正确内容，但排序不稳定 | 语料很小，或初次召回已经满足指标 | 增加延迟、分数被误解为概率、无法挽回漏召回 | 中高/中高/中 | 混合搜索、调整召回 | Day 1 明确 `Query + 候选块 → 排序`；待 Day 6 实测 |
