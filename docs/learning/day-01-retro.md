## 最重要的理解

今日形成的关键理解来自学习者原始表达：

> 在创建文档向量数据库的时候，需要对文档切块之后的内容做Embedding，存在向量数据库，RAG检索的时候，对query做Embedding之后，用于向量数据库的查询，之后检索出来top-n发送到reranker做精排，得到top 5之类的结果

补充后的完整认识是：Embedding 支持粗召回，Reranker 负责候选精排，RAG 是检索、上下文增强和 LLM 生成的整体模式；最终效果由模型层、应用层和工程层共同决定。

## 决策题修订

1. 开始前答案：LLM 输入输出是 Token；误认为 Embedding 输出 Token；不知道 Reranker。
   结束后答案：LLM 生成 Token；Embedding 输出数值向量；Reranker 输入 Query 和候选文本，输出相关性分数或排序。
2. 开始前答案：不知道。
   结束后答案：RAG 改变推理时的外部上下文；Tool Calling 扩展可请求的外部动作；微调通过训练改变模型参数和稳定行为。
3. 开始前答案：不知道。
   结束后答案：应用效果由模型、Prompt、Context、检索、工具、校验、评测、延迟和成本共同产生，更大的模型不保证整条链路更好。

## 失败案例

- 输入或场景：从 Atlas 会议记录提取行动项，并只输出 JSON 数组。
- 预期：只提取两个明确的未来行动项，不添加 Markdown 围栏。
- 实际：Claude 两次都把“确认既有上线目标”额外识别为行动项，并添加代码围栏；代码解释也两次超过 300 字限制。
- 所属层：模型 / 应用 / Eval。
- 根因假设：任务边界和格式约束对模型具有区分度；自然语言指令不能替代 Schema 校验和回归评测。
- 下一次验证：增加冲突信息、修正信息、无答案和严格结构化输出的 Challenge Set。
- 是否加入回归评测：是。

## 效率与质量

- 原人工耗时：未建立基线，不编造数据。
- 当前耗时：16 次模型调用平均约 9.4–10.9 秒/次，不含人工评分。
- 自动化节省：运行器自动调度、计时、记录 Token、保存原始输出和生成评分表。
