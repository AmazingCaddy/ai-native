## Q1：LLM、Embedding 与 Reranker 的输入和输出是什么？

### 问题

大语言模型、Embedding 模型和 Reranker 的输入、输出分别是什么？

### 原始回答

> 大语言模型的输入和输出都是token，embedding模型的输入是string，输出是token. Rank的输入和输出我不知道

### 判断

部分正确。

- LLM 部分抓住了模型内部处理 Token 序列这一点。
- Embedding 的输入可以是文本等内容，但输出不是 Token，而是数值向量。
- Reranker 部分尚未掌握。

### 标准答案与解释

- LLM：输入 Token 序列，输出生成的 Token 序列，常用于生成、总结和推理。
- Embedding 模型：输入文本、图片等内容，输出固定维度的数值向量，常用于语义搜索、聚类和 RAG。
- Reranker：输入一个查询和若干候选文档，输出候选的相关性分数或重新排序后的结果。

### 易错点

不要把 Token 和 Embedding 向量混为一谈。Token 是模型处理的离散文本单位；Embedding 是表示语义位置的一串数值。

### 一句话记忆

LLM 生成内容，Embedding 寻找相似内容，Reranker 给候选内容重新排队。

## Q2：RAG、Tool Calling 与微调改变什么？

### 问题

RAG、Tool Calling 和微调分别改变 AI 系统的哪一部分？

### 原始回答

### 判断

## Q3：为什么模型更大不等于应用效果更好？

### 问题

为什么“模型更大”不代表“应用效果一定更好”？

### 原始回答

> 不知道

### 判断

待修正。

### 标准答案与解释

AI 应用的最终效果不仅取决于模型能力，还取决于输入资料、Prompt、检索质量、工具可靠性、输出校验、工作流以及成本、延迟和隐私约束。更大的模型也可能更慢、更贵；对于简单且约束明确的任务，小模型可能更稳定、更经济。

### 易错点

不要只比较模型排行榜。应用质量是整个系统共同产生的结果，必须使用真实任务和评测集验证。

### 一句话记忆

模型只是系统的一部分，应用效果取决于整条链路。

## Q4：三类模型在文档问答中的分工

### 问题

用户问“公司的年假怎么申请？”，系统需要从 1000 份内部文档中找到依据并回答。LLM、Embedding 和 Reranker 分别负责什么？

### 原始回答

> LLM负责推理回答问题，Embedding负责把输入的问题转换成向量，用于检索。LLM和Embedding检索的结果，总的按照reranker排序

### 判断

部分正确，已经掌握了三者的大体分工。

- LLM 负责基于找到的资料组织和生成回答，这一点正确。
- Embedding 不只转换问题，也要预先转换文档或文档片段，才能计算语义相似度并检索候选内容。
- 通常不是 LLM 和 Embedding 分别产生检索结果。向量检索先产生候选，Reranker 再结合原始问题对候选重新评分，最后才交给 LLM。

### 标准答案与解释

1. 文档入库时，Embedding 模型把文档片段转换为向量并保存。
2. 用户提问时，同一个或兼容的 Embedding 模型把问题转换为向量。
3. 向量检索根据问题向量与文档向量的相似度找出一批候选片段。
4. Reranker 同时查看原始问题和候选片段，给候选重新评分并选出最相关的内容。
5. LLM 根据排名靠前的资料生成回答，并应标注引用；资料不足时应该拒绝猜测。

### 易错点

不要把“生成”和“检索”混成同一步。LLM 在基础 RAG 流程里主要消费检索结果并生成回答；检索通常由搜索系统完成。

### 一句话记忆

Embedding 召回候选，Reranker 精排候选，LLM 阅读依据并作答。

## Q5：频繁变化的知识应该使用 RAG 还是微调？

### 问题

公司的请假制度每周都可能变化，希望 AI 始终根据最新制度回答。应该优先选择 RAG 还是微调？为什么？

### 原始回答

> 是微调吗？我是猜测的，因为频繁变化意味着向量数据库也得频繁重建吗

### 判断

待修正，但追问抓住了一个重要的工程成本问题：知识变化后，索引究竟需要更新多少。

### 标准答案与解释

应该优先选择 RAG。

请假制度属于频繁变化、需要来源依据的外部知识。RAG 可以在制度更新后替换原文档，并只删除或重新计算受影响的文档片段向量；通常不需要重建整个向量数据库。

完整重建通常发生在更换 Embedding 模型、修改全局切分策略或元数据结构、需要重新处理全部文档等情况下。日常新增、修改或删除少量制度文档，可以使用增量 `upsert` 和删除完成。

微调会修改模型参数。每周通过微调更新事实，需要反复准备数据、训练和评测，还难以保证旧规定被彻底遗忘，也不便于给出可核验引用。因此微调更适合稳定的行为模式、输出风格、格式或专项任务，而不是频繁变化的事实库。

两者也可以组合：使用微调稳定行为，使用 RAG 提供最新知识。

### 易错点

不要把“更新向量索引”理解成“重建全部向量数据库”。还要区分知识和行为：知识频繁变化通常优先更新外部资料；希望模型长期稳定地按照某种方式行动，才更可能考虑微调。

### 一句话记忆

变化的事实放在 RAG，稳定的行为才考虑微调；文档变化通常只增量更新相关向量。

## Q6：Token 与 Context Window 如何影响应用？

### 问题

模型看到“AI Native 工程师”这段文本时，Token 和 Context Window 分别是什么？如果一次对话超出 Context Window，可能发生什么？

### 原始回答

> Token 是 "AI", "Native" "工程师" 三个。context window是什么，我不知道

### 判断

部分正确。已经理解 Token 是文本经过切分后的单位，但不能根据肉眼看到的单词和空格断定具体 Token 数量。

### 标准答案与解释

Token 是模型 tokenizer 将输入转换得到的离散编号单位。它可能对应一个单词、单词片段、汉字、标点或空格组合。不同模型使用的 tokenizer 不同，因此“AI Native 工程师”不一定恰好是三个 Token，必须使用对应模型的 tokenizer 或 API 用量信息确认。

Context Window 是模型在一次请求中能够处理的 Token 总容量。它通常包含系统指令、用户消息、历史对话、工具结果、检索资料等输入；生成输出也需要占用可用预算，具体限制和计算方式取决于模型及 API。

当内容超出窗口时，常见结果包括：API 拒绝请求；应用主动截断或删除较早内容；应用先总结旧内容；或者减少留给输出的空间。如果重要指令或事实被移出窗口，模型就无法直接使用它们，可能出现遗忘、矛盾或答非所问。

### 易错点

不要把 Token 简单等同于单词，也不要把 Context Window 理解为模型的永久记忆。窗口大只表示一次能看到的内容更多，并不保证模型能同等准确地利用窗口中的每个细节。

### 一句话记忆

Token 是模型读写的计量单位；Context Window 是一次请求能装下的 Token 容量，不是永久记忆。

## Q7：Transformer 与 Attention 解决什么问题？

### 问题

阅读句子“我把书放在桌子上，因为它很稳固”时，模型需要判断“它”更可能指什么。你认为 Attention 在这里起什么作用？Transformer 又是什么？

### 原始回答

> 不懂诶

### 判断

待修正。目前还没有建立 Attention 和 Transformer 的概念。

### 参考答案与解释

句子中的“它”更可能指“桌子”，因为“稳固”通常用于描述支撑书的桌子。这个句子仍可能有歧义，模型需要结合上下文学习词与词之间的关系。

Self-Attention 让句子中的每个 Token 根据当前任务，为其他 Token 分配不同权重并汇总信息。处理“它”时，模型可以让“桌子”获得比“书”等词更高的相关权重，使“它”的上下文表示包含更多关于“桌子”的信息。多头 Attention 可以同时学习指代、语法、位置等不同关系。

Transformer 是以 Attention 为核心搭建的神经网络架构。典型 Transformer 层还包含前馈网络、残差连接、归一化和位置信息，并将多层重复堆叠。现代 LLM 通常基于 Transformer；Transformer 是架构，LLM 是使用大量数据训练出来的语言模型。

Transformer 的一个重要优势是可以并行处理序列中的多个 Token，并更直接地建立远距离 Token 之间的联系。生成式模型通常使用因果 Attention，使当前位置只能查看它之前的 Token。

### 适用边界与易错点

Attention 不等于“模型已经理解”，也不能保证正确解决所有歧义。Attention 权重只是模型内部信息混合的一部分，不能简单当作模型答案的完整解释。

### 一句话记忆

Attention 决定当前 Token 应重点参考哪些 Token；Transformer 把这种机制堆叠成处理上下文的网络架构。

## Q8：模型层、应用层和工程层如何分工？

### 问题

请尝试把下面这些能力分到“模型层、应用层、工程层”：LLM、Embedding 模型、RAG、Tool Calling、Prompt、重试、日志、评测。每一层主要负责什么？

### 原始回答

> 模型层：LLM、Embedding 模型。负责生成答案
> 应用层：RAG、Tool Calling、Prompt。负责给LLM提供更加精确的信息
> 工程层：重试、日志、评测。负责提升整个系统的健壮性

### 判断

部分正确。三个分类全部正确，工程层的概括也基本准确；模型层和应用层的职责需要进一步区分。

### 参考答案与解释

- **模型层**提供训练得到的基础能力。LLM 生成或理解 Token；Embedding 模型输出数值向量，用于相似度计算，它不生成答案。
- **应用层**把模型能力组织成具体产品流程。Prompt 提供任务指令和上下文；RAG 检索外部知识；Tool Calling 让模型提出对外部系统的读取或执行请求。它不仅让信息更精确，也定义任务如何完成。
- **工程层**让系统能够可靠运行和持续改进。重试处理暂时性失败；日志让问题可观察、可诊断；评测衡量系统是否真的变好。生产系统通常还会在这一层处理安全、成本、延迟、缓存和降级。

### 适用边界与易错点

这三层是帮助分析系统的概念边界，不一定对应三个独立部署服务。某些能力也会跨层，例如评测结果会反过来驱动 Prompt、RAG 和模型选择的改进。

### 一句话记忆

模型层提供能力，应用层组织任务，工程层保证系统可靠、可测、可运营。

## Q9：如何公平比较两个模型？

### 问题

现在要比较模型 A 和模型 B 在“从会议记录提取行动项”任务上的效果。你会怎样设计一个公平、可重复的实验？需要固定什么条件、记录哪些指标？为什么不能只运行一次就下结论？

### 原始回答

> 不懂，请告诉我答案

### 判断

待修正。目前还没有建立模型评测实验的基本框架。

### 参考答案与解释

一个公平实验应当先定义任务和成功标准，再准备一组具有代表性的固定评测样本及参考答案。比较时只改变模型，保持输入、系统指令、Prompt、输出 Schema、可用工具、检索资料以及温度、最大输出等配置尽可能一致。

对于“会议记录提取行动项”，可以记录：行动项正确率和召回率、负责人/截止时间等字段正确率、Schema 通过率、遗漏和幻觉数量、延迟、输入/输出 Token、成本、API 失败率以及人工可用性评分。

每个模型应在全部评测样本上运行；对于具有随机性的配置，重要样本还应重复多次。记录模型名称和具体版本、运行日期、Prompt 版本、数据集版本、完整参数、原始输出和失败案例。比较平均结果、波动和不同类型样本上的表现，而不只看总分。

不能只运行一次，因为单个样本可能不具代表性，模型输出可能存在随机波动，API 也可能出现暂时性错误。即使温度为 0，托管模型也不一定保证跨时间完全相同的输出。

### 适用边界与易错点

“公平”不是只给两个模型同一句 Prompt：如果某模型要求不同的最佳实践，还可以补充一轮“各自优化配置”的比较，但必须把“统一条件”和“各自最优”两类实验分开报告。自动指标也不能完全替代人工检查失败案例。

### 一句话记忆

固定任务、数据和条件，只改变模型；同时测质量、速度、成本和波动，并保存完整实验记录。

## Q10：第一天应该怎样启动主线项目？

### 问题

主线项目是“团队知识与任务助手”。在 Day 1 只允许建立最小项目骨架，不实现 RAG、Agent 或复杂界面。你认为今天至少应该确定哪些成功标准、准备哪些样例数据，并建立什么最小可运行验证？

### 原始回答

> 成功标准：团队知识召回率大于xx%，任务分配的owner正确率。
> 样例数据：没什么想法。
> 最小可运行验证：也没什么想法

### 判断

部分正确。提出了两个可量化的未来质量指标，但还需要区分“Day 1 完成标准”和“项目最终质量指标”。

### 参考答案与解释

“知识召回率”和“Owner 正确率”适合作为后续质量指标，但必须先定义评测集、相关文档和正确 Owner，不能只保留 `xx%`。Day 1 暂时不实现检索和任务分配，因此今天的成功标准应聚焦于项目是否已经被清楚定义并能重复启动。

Day 1 至少完成：明确目标用户、三个核心用例和不做的内容；保存 10 个真实问题，其中至少 3 个在资料中没有答案；写下未来指标定义；建立应用和测试目录、项目 README、`.env.example` 与凭据忽略规则；提供一条可重复的启动命令和测试命令。

样例数据可以准备 5 至 10 份允许用于实验的团队文档、3 份经过脱敏的会议记录、10 个真实问题，以及每份会议记录对应的人工标注行动项。没有真实资料时，可以先编写小型合成数据，但必须明确标记，后续再补充真实样本。

最小可运行验证不需要调用模型：启动一个 CLI 或 Web/API 骨架，执行健康检查或 `Hello World`，并运行一个自动测试验证返回结果。换一台机器按照 README 也能完成安装、启动和测试，才说明骨架可复现。

### 适用边界与易错点

不要在 Day 1 就实现 RAG、Agent 或复杂界面。指标必须带有清晰定义和可计算的标注数据；“召回率”还要说明是检索 Recall@K、引用覆盖率还是答案正确率，它们不是同一个指标。

### 一句话记忆

第一天先定义边界、准备可评测样本，并用一次启动和一个测试证明项目骨架可复现。

## MAP-Q01：Embedding、RAG 和 Reranker 的先后关系

### 问题

请用一句话描述 Embedding、RAG 和 Reranker 在文档问答中的先后关系。

### 原始回答

> 在创建文档向量数据库的时候，需要对文档切块之后的内容做Embedding，存在向量数据库，RAG检索的时候，对query做Embedding之后，用于向量数据库的查询，之后检索出来top-n发送到reranker做精排，得到top 5之类的结果

### 判断

正确。回答准确描述了离线建库与在线检索，并正确给出了向量召回、Reranker 精排的先后关系。

### 参考答案与解释

离线阶段，文档经过解析和切块后，由 Embedding 模型转换为向量并写入向量索引。在线阶段，Query 使用兼容的 Embedding 模型转换到同一向量空间，先从向量数据库召回 Top-N 候选文本块；Reranker 再同时考察 Query 和每个候选块，重新计算相关性顺序，选出 Top-K 结果作为上下文交给 LLM 生成答案。

RAG 是包含“检索、上下文组装、生成”的完整模式，不应只等同于向量数据库查询。Top-N 和 Top-K 也不是固定值，需要根据召回率、精度、延迟、成本和上下文窗口进行评测选择。

### 适用边界与易错点

- 文档块与 Query 的向量必须位于兼容的向量空间，通常使用同一 Embedding 模型或明确支持非对称检索的一对模型。
- 初次召回不一定只能使用向量检索，也可以使用关键词检索或混合检索。
- Reranker 通常只处理召回后的候选集，不适合直接遍历整个文档库。
- 精排结果还需要携带原文和来源信息，而不能只把相关性分数发送给 LLM。

### 一句话记忆

Embedding 负责粗召回，Reranker 负责精排序，RAG 把检索结果送入 LLM 完成增强生成。

### 追问 1：RAG 是否指整个完整过程？

#### 原始追问

> 所以RAG指的是整个完整的过程对吧

#### 回答

对。RAG 是 Retrieval-Augmented Generation（检索增强生成），核心在线流程包含：检索（Retrieval）、把检索结果加入模型上下文（Augmentation）以及让 LLM 基于增强后的上下文生成答案（Generation）。

工程语境中还需要区分两个范围：

- **RAG 在线流程**：从用户 Query 开始，到检索、精排、上下文组装和 LLM 生成答案。
- **RAG 系统或 RAG Pipeline**：通常还包括文档解析、清洗、切块、Embedding、索引构建和增量更新等离线准备工作。

因此，把“RAG”只说成“去向量数据库搜索”是不完整的；但日常交流中有人会用“RAG 检索”简称其中的检索阶段，需要结合上下文判断。

## CHECK-Q01：技术地图口述验收

### 问题

请用不超过五句话说明：模型层、应用层和工程层分别负责什么，以及 Embedding、RAG、Reranker、LLM 和 Eval 如何连接起来。

### 原始回答

> 模型层提供基础能力，应用层将模型，上下文，知识和动作组织成具体任务，工程层让系统可靠，可测，可诊断，可运营。
> 离线：文档切块做embedding，存储到向量数据库
> 在线：query做embedding之后进入向量数据库检索出topN，传递给reranker做精细化排序得到topK，再作为上下文送给LLM，对每次query的结果做检测和打分就是eval

### 判断

正确，口述验收通过。回答准确说明了三层职责，并完整串联了文档切块、Embedding、向量召回、Reranker、上下文和 LLM。

### 参考答案与补充

模型层提供生成、Embedding 和 Reranker 等基础能力；应用层用 Prompt、Context、RAG、工具和工作流把能力组织成具体任务；工程层通过评测、日志、重试、安全与成本控制，让系统可靠、可测、可诊断、可运营。

离线阶段将文档解析、切块并建立向量索引；在线阶段将 Query 转为向量，召回 Top-N 候选，经 Reranker 精排为 Top-K，再组装进上下文交给 LLM 生成答案。

需要补充的边界是：Eval 不只对每次 Query 的最终答案打分。它既可以离线批量运行评测集，也可以在线记录反馈；还能分别测量检索召回、排序、答案正确性、引用、工具调用、延迟、Token 和成本。

### 一句话记忆

模型层给能力，应用层组任务，工程层保质量；Eval 测整条链路，而不只测最终答案。
