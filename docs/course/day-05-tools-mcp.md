## 核心内容

### Tool Calling 循环

1. 应用把允许使用的工具及 Schema 发给模型。
2. 模型返回工具名和候选参数，而不是实际执行结果。
3. 应用校验工具、参数、用户权限和当前状态。
4. 写操作先创建待批准记录；读操作也需限制数据范围。
5. 应用执行工具并保存审计记录。
6. 裁剪和标记工具结果后再传回模型。
7. 模型根据可信工具结果生成最终回答。

### MCP 的位置

MCP 为客户端发现和调用外部工具、资源及提示提供通用协议。它适合多个 AI 客户端共享同一集成，或者集成需要独立部署和权限边界的情况。只有单个应用内部的两个函数时，普通接口通常更简单。

### 安全要求

- 工具列表使用显式 Allowlist
- 参数执行运行时校验，不能只依赖模型生成的 JSON
- 读操作限制资源范围，写操作默认需要确认
- 工具结果视为不可信输入，防止其中包含注入指令
- 设置每次任务的最大工具调用次数和总时限
- 审计记录包含调用者、输入摘要、批准者、结果和时间

## 动手实验：同一能力的三种暴露方式

选一个简单能力，例如搜索本地文档。分别描述：

1. 业务代码直接调用函数
2. 通过内部 HTTP API 调用
3. 通过 MCP Server 暴露

比较部署、复用、鉴权、调试、延迟和维护成本。只需实现最适合当前项目的一种；MCP 实现可作为选做。

## 主线项目任务

1. 实现只读工具 `search_documents` 或等价能力。
2. 实现写工具 `create_task_draft`，默认只创建草稿或模拟结果。
3. 为工具定义严格 Schema 和业务校验。
4. 写工具必须经过 Preview 和 Approve 两个阶段。
5. 为未知工具、无效参数、无权限、超时和重复执行添加测试。
6. 将 ToolRun 保存到持久层，并对长结果做长度限制。
7. 为整个循环设置最大步骤数，例如 5 次。
