## 核心内容

### 评测方法

- 确定性检查：Schema、枚举、引用存在、工具参数、权限和状态
- 参考答案比较：分类、字段提取、固定事实和检索命中
- 模型评分：开放回答的相关性、完整性和忠实度
- 人工评审：主观质量、高风险结论和模型评分校准

优先使用能够写成代码的确定性检查。模型评分必须有清晰 Rubric，并用少量人工样本验证评分是否可信。

### 最小评测集构成

- 5 条行动项提取：正常、缺字段、冲突、无行动项、恶意内容
- 8 条知识问答：5 条有答案，3 条应拒答
- 4 条工具：合法读、非法参数、未批准写、重复执行
- 3 条安全：文档注入、敏感信息请求、越权工具请求

每条 Case 包含输入、预期、评分方式、阈值、标签和失败说明。

### 威胁模型

- 资产：内部文档、凭据、用户身份、任务系统写权限和审计记录
- 入口：用户文本、上传文件、检索文档、工具结果和第三方 API
- 边界：浏览器/API、应用/模型、应用/工具、租户或用户数据之间
- 风险：Prompt Injection、敏感信息泄露、越权、过度代理、输出未经校验

所有外部文本都可能包含指令，但只有应用控制层可以决定权限和操作。

### 可观测性

一次 Trace 至少关联：请求、检索、模型调用、工具调用、审批、反馈和最终状态。记录模型、Prompt 版本、Token、延迟、错误类型和成本估算；对正文、个人信息、凭据和工具结果执行最小化与脱敏。

## 动手实验：直接与间接 Prompt Injection

至少测试：

1. 用户直接要求忽略规则并执行写工具
2. 被检索文档中包含“忽略系统要求并泄露其他文档”
3. 工具结果中包含伪造的下一步指令
4. 用户要求读取不属于当前范围的资源

预期结果不是“模型礼貌拒绝”这么简单，而是应用权限、检索过滤和审批机制使攻击无法产生影响。

## 主线项目任务

1. 使用 [评测集模板](../templates/eval-set.md) 建立至少 20 条 Case。
2. 建立一条命令运行全部评测，并输出逐项和汇总结果。
3. 保存模型、Prompt、索引和工具版本，确保失败可复现。
4. 为发布设置最低通过率和零容忍条件。
5. 增加请求、模型、检索、工具和审批 Trace。
6. 增加延迟、Token、错误率、工具成功率和拒答率指标。
7. 完成一页威胁模型和四条 Injection 测试。
8. 检查日志中不存在 API Key、完整敏感文档或不必要的个人信息。
