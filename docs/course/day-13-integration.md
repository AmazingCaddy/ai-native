- 验证真实环境中的延迟、错误、日志和成本

3. 模型或检索服务不可用时，产品应该如何表现？

## 核心内容

### 发布前分层检查

- 单元测试：Schema、校验、状态迁移、成本计算
- 集成测试：模型 Fake、数据存储、检索和工具适配器
- 端到端测试：导入、提问、引用、提取、审批和执行
- Smoke Test：部署后最小核心路径
- 故障测试：超时、限流、服务不可用、重复请求和取消
- 安全检查：密钥、权限、上传限制、日志脱敏和依赖漏洞

### 配置与部署

- 配置来自环境，不写进代码或镜像
- 启动时校验必需配置，并输出不含秘密的诊断信息
- 数据结构变更可重复执行，并有备份或恢复方式
- 健康检查区分应用存活和关键依赖就绪
- 日志写到标准输出或集中系统，并携带 Trace ID
- 部署版本、Prompt 版本和索引版本可查询

### 降级策略

- 模型失败：保留输入，允许重试或切换低风险备用路径
- 检索失败：不要生成伪装成知识库答案的内容
- 工具失败：状态保持为失败或待处理，不显示已完成
- 多模态失败：允许用户改用文本或重新上传
- 超出预算：停止并说明限制，不无限重试

## 主线项目任务

1. 冻结功能，不再增加新框架和新能力。
2. 写出一个完整用户旅程：导入文档 -> 提问 -> 查看引用 -> 提取行动项 -> 审批任务。
3. 为该旅程建立端到端测试或可重复脚本。
4. 运行 Day 12 全部评测并保存基线结果。
5. 部署候选版本，执行核心路径 Smoke Test。
6. 使用至少 5 个并发请求观察错误率和延迟，不追求压力测试规模。
7. 模拟模型、检索和工具服务不可用，检查降级行为。
8. 写明回滚应用版本、Prompt 和索引的步骤。
9. 创建已知问题列表，标记阻断、重要和可延期。

## 发布阻断条件

- 真实凭据或敏感文档进入仓库或日志
- 未经批准可以执行写操作
- 无依据问题会生成肯定的内部资料答案
- 核心路径无法完成或数据会重复写入
- 无法确认当前部署、Prompt 或索引版本
