# Day 10：团队协作与治理

## 今日目标

- 找出团队中适合 AI 的高频工作，而不是先寻找新工具
- 把个人技巧转成可复用、可审查的团队 SOP
- 把一条验证有效的流程封装成可维护的团队 Skill
- 建立数据、权限、评测和责任边界
- 上线至少一条能被同事实际试用的工作流

## 开始前决策题

1. 个人使用有效的 Prompt，怎样才能成为团队能力？
2. 团队采用 AI 后，哪些指标比“生成了多少内容”更重要？
3. 谁对 AI 生成并最终发布的结果负责？

## 核心内容

### 工作流盘点方法

为每个候选流程记录：频率、单次耗时、参与人数、输入稳定性、输出可验证性、错误影响、数据敏感度和现有系统集成难度。

优先选择高频、低风险、容易验证的流程，例如：

- Issue 到实施计划草稿
- PR 摘要和 Review 检查
- 会议记录到行动项草稿
- 周报或项目状态草稿
- 文档问答和新成员入门

### 团队资产

- 仓库规则和完成标准
- 经过评测的 Prompt 与 Schema
- 有明确触发条件、Owner 和版本的共享 Skill
- 共享工具和 MCP 集成
- 示例输入、黄金输出和失败案例
- 数据分类、权限与审批规则
- 负责人、版本、变更记录和下线方式

### 采用指标

- 完成相同任务节省的人工时间
- 一次通过率和人工修改量
- 漏检、误报和回滚次数
- 用户主动复用率
- 高风险操作的人工确认覆盖率

## 动手任务：团队 AI 工作流清单

至少评估以下三个流程：

1. 开发：Issue -> 计划 -> 实现 -> 测试 -> Review
2. 协作：会议记录 -> 行动项 -> 审批 -> 任务草稿
3. 汇报：数据和更新 -> 周报草稿 -> 事实核对 -> 发布

为每个流程写一页卡片：用户、触发、输入、输出、模型步骤、确定性步骤、人工检查、数据等级、成功指标和停止条件。选择其中一个进入真实试用。

## 主线项目任务

1. 为团队助手增加清晰的用户入口和示例。
2. 写一份不超过两页的团队使用 SOP。
3. 标注允许上传、禁止上传和需要批准的数据类型。
4. 明确生成草稿、批准操作和发布结果的责任人。
5. 将一条已验证流程封装成团队 Skill，包含触发条件、步骤、参考资料和完成标准。
6. 为共享 Prompt、Schema、Skill 和工具添加 Owner 与版本。
7. 增加反馈入口，并将反馈关联到 Trace 或具体结果。
8. 收集一位同事的完成时间、修改量和理解障碍；无法邀请同事时，由自己按新用户视角完整走查。
