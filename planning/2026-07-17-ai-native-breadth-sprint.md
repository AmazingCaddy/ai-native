# AI Native 两周广度冲刺计划

> 配套的每日课程、资料索引和练习模板见 [AI Native 14 天课程包](../docs/intro.md)。

## 目标

- 周期：14 天
- 投入：每天约 6 小时，共约 84 小时
- 起点：软件工程师
- 目标：建立完整的 AI 技术认知地图，能够判断不同技术的用途；提升个人与团队工作效率；独立开发并上线一个 AI 应用。

两周的目标不是精通所有方向，而是建立技术判断力，并用一个完整项目验证关键概念。

## 主线项目

贯穿两周开发一个“团队知识与任务助手”：

- 导入团队文档
- 基于资料回答问题并标注来源
- 生成结构化任务
- 调用外部工具
- 高风险操作需要人工确认
- 记录评测、日志和成本
- 最终部署上线

优先使用熟悉的编程语言和 Web 框架，避免将时间消耗在学习新框架上。

## 每日节奏

| 环节 | 时间 | 内容 |
| --- | ---: | --- |
| 概念与案例 | 1.5 小时 | 学习当天主题的原理、适用场景和边界 |
| 项目开发 | 3.5 小时 | 将当天技术集成到主线项目 |
| 测试与评测 | 0.5 小时 | 验证效果并记录失败案例 |
| 地图与复盘 | 0.5 小时 | 更新技术地图和判断矩阵 |

每天看资料不超过两小时。每个概念都必须通过代码、实验或真实工作流验证。

## 14 天安排

| 天 | 主题 | 学习重点 | 当日产出 |
| ---: | --- | --- | --- |
| 1 | AI 技术全景 | 模型类型、核心能力、局限与应用分层 | 第一版 AI 技术地图 |
| 2 | Prompt 与 Context | 指令设计、上下文组织、结构化输出、模型选择 | Prompt 模板与模型对比实验 |
| 3 | AI 编程工作流 | 需求澄清、任务拆分、编码、测试、Review | 一套可重复使用的 AI 开发工作流 |
| 4 | 模型 API | 流式响应、结构化数据、重试、超时与错误处理 | 可用的聊天接口 |
| 5 | Tool Calling 与 MCP | 工具协议、参数校验、权限和执行结果回传 | 能调用一个真实工具的助手 |
| 6 | RAG 与搜索 | Embedding、切分、向量搜索、关键词搜索、混合检索 | 带引用的文档问答功能 |
| 7 | 多模态 | 图片、OCR、语音、复杂文档理解 | 图片或复杂文档信息提取功能 |
| 8 | 工作流自动化 | 触发器、步骤编排、失败恢复与幂等 | 自动化一项真实的个人重复工作 |

| 10 | 团队协作 | 会议、任务、周报、知识共享和审批场景 | 一项可供团队使用的自动化流程 |

## 技术判断矩阵

每天遇到一种技术，都记录以下问题：

1. 它解决什么问题？
2. 输入和输出是什么？
3. 什么时候应该使用？
4. 什么时候不应该使用？
5. 主要失败模式是什么？
6. 成本、延迟和隐私影响如何？
7. 有哪些替代方案？

重点掌握以下取舍：

- Prompt、RAG 与微调
- 长上下文、向量检索与混合检索
- 固定工作流、单 Agent 与多 Agent
- Tool Calling、MCP 与普通 API
- 云端模型与本地模型
- 大模型、小模型与模型路由
- 自动执行与人工确认

## 每日完成标准

每天结束前必须完成：

- 一个可以运行的代码或工作流增量
- 至少一个失败案例及原因记录
- 技术地图新增或修正一个节点
- 判断矩阵新增至少一项技术
- 提交一条简短的当日复盘

## 最终验收标准

第 14 天应当拥有：

- 一张覆盖至少 25 个概念的 AI 技术地图
- 一份可以指导技术选型的判断矩阵
- 三个已经投入使用的个人或团队 AI 工作流
- 一个独立开发并部署的 AI 应用
- 至少 20 条自动评测用例
- 一个明确的下一阶段深入方向

## 下一阶段选择

根据两周内的项目体验、评测结果和兴趣，从以下方向选择一到两个继续深入：

- AI 应用工程
- Agent 与自动化
- AI 产品与用户体验
- 模型、数据与评测
- AI 基础设施与平台工程
